## Where the project stands (as of 2026-05-30, ~5 focused days)

### Engine — solid and validated
- 5-layer architecture: Geometry/Topology → Quantities/MEP/Validation → BOQ
  presentation → UI/Persistence/Export.
- **34 verify scripts, all green.**
- Referential integrity (`verifyIntegrity` INV-W1–W10) gates every state-building
  test.

### Draw + Room Tool — canvas-validated today
- Inside-face / Center / Outside-face draw modes.
- Auto-focusing length input (chain start + per segment).
- Single room tool (`room_detect`, key **R**) — face-graph based, handles
  sub-span rooms on T-junctioned full-length walls.

### BBS engine — validated against professional reference
- IS 2502 RebarGroup engine. Categories: footing, strap footing, sub/super
  column, column, tie/plinth/lintel/roof/generic beam, sunshade, loft, staircase,
  slab variants (`BBS_CATEGORY` ≈ 16 entries incl. aliases). This **matches the
  reference workbook's 11 categories** — the engine taxonomy is a superset, the
  "11" is the Karthick workbook's count, not the engine's.
- Two modes: **IS_STRICT** (code-exact, byte-identical to original) and
  **SITE_PRACTICE** (matches a contractor's hand BBS to **±2%/bar**).
- Three output levels: per-bar detail, category abstract, Excel + PDF.
- **Excel/PDF export is shipped AND wired** in `BBSSchedulePanel.jsx` (buttons
  call `exportBbsExcel`/`exportBbsPdf`; `src/export/bbs.js` + `verify-bbs-export`
  green). The `// TODO BBS-5b` comment and the CLAUDE.md BBS-5b backlog line are
  **stale** — the feature exists; it's clickable.
- Validated against the Karthick M-City BBS workbook (`docs/BBS-VALIDATION-KARTHICK.md`).

### BOQ engine — working, quantities solid, pricing partial
- Single canonical `computeBoqPresentationModel` is the only source for Excel +
  PDF totals; `verify-boq` byte-equality canary locks the pipeline.
- Export is **bucket-driven** (`src/export/_buckets.js`, **~22 category
  buckets**); the number of Excel sheets varies per project (Summary + non-empty
  buckets + Raw Data) — it is **not** a fixed sheet list.
- **Pricing exists but is not persistent:** the BOQ panel has live, ephemeral
  rate inputs + a **Project Cost Summary** (labor / supervision / overhead /
  profit / GST). The gap is a **persistent rate library / catalog**, not the
  ability to price.
- Masonry aggregates by `materialKey`, **not by wall thickness** (9" structural
  and 4.5" partition of the same material share one bucket — no thickness split).
  *(This is the one repo-confirmed item from my afternoon BOQ-delta review.)*

### Locked architectural rules (CLAUDE.md — do not violate)
1. **Phase W**: one wall = one full-length entity, T-junctions first-class. NO
   auto-split, NO manual split.
2. **RebarGroup is computed, never persisted.**
3. **No magic numbers in BBS generators** — all IS factors in the IS 2502 catalog
   (`src/specs/cuttingLength.js`), per-project overridable.
4. **Single source of truth over layered fallbacks.**
5. **Verify must drive the actual UI entry-point** with realistic state.
6. **MCP-first** for framework patterns; cited web research for IS-code /
   Chennai-practice rules.
7. **Greenfield**: no migration code, no shims — delete, don't deprecate.
